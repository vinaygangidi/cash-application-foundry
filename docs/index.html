Background
What is Cash Application?
Cash Application is the Accounts Receivable process of matching incoming bank payments to open invoices and posting the results to an ERP (SAP, Oracle, NetSuite). It sounds simple — but in practice it's one of the most labor-intensive and error-prone processes in enterprise finance.

❌ The Manual Problem

  • 💸 Customer pays $29,250 but invoice is $29,500 — was it a freight deduction, a short pay, or a dispute?
  • 🏦 SWIFT wire shows payer as "GREENFIELD TECH SOLUT" — name is truncated at 35 chars. Who is this?
  • 🔄 Payment comes from "ACE Capital" but your customer is "Riverside Mfg" — factoring company bought the invoice.
  • ⚖️ A payment arrives for an invoice in active litigation — posting it would be a compliance violation.
  • 💱 EUR payment arrives, FX conversion needed — manual calculation required to reconcile the USD amount.
  • ⏰ An AR analyst handles 8–10 edge cases per hour. These stack up to hundreds daily.

✅ What Our Pipeline Does

  • 🏦 Parses the bank statement, normalizes truncated names, detects flags like NSF, FX, duplicate payments.
  • 📒 Enriches the AR ledger with aging buckets, dispute status, collection risk, credit holds.
  • ⚖️ Uses Code Interpreter to run exact Python arithmetic — verifies multi-invoice sums, FX conversions.
  • 🧠 AI reasoning agent categorizes every mismatch: deduction, short pay, dispute, factoring, etc.
  • ✅ Generates ERP-ready GL journal entries and posting batches with full audit trail.
  • ⚡ Streams results to the browser in real-time via SSE — 35 cases resolved in under 60 seconds.
Core Concept
The Reconciliation Process
Reconciliation means proving that every dollar in the bank statement corresponds to a specific invoice (or set of invoices) in the AR ledger — and documenting exactly why any difference exists.

1

Receive & Parse the Bank Statement

The bank delivers a daily statement listing every incoming wire, ACH, and check. Each entry has a date, amount, payer name, and sometimes a remittance reference. The payer name is often truncated (SWIFT limits to 35 chars), misspelled, or belongs to a third party like a factoring company.

BankStatementIntelligenceAgent
2

Load & Enrich the Open AR Ledger

The open AR ledger lists every unpaid invoice: customer, amount, due date, and aging. The AR agent enriches this with risk flags — which invoices are disputed, in legal hold, past due, on credit hold, or linked to a factoring arrangement.

ARLedgerAgent
3

Match Payments to Invoices

Each bank payment is matched against open invoices using fuzzy name matching, remittance reference detection, and exact arithmetic. Multi-invoice payments are split across invoices. The Code Interpreter runs Python to verify that the numbers add up exactly — critical for catching rounding errors and FX discrepancies.

ReconciliationAgent Code Interpreter
4

Reason About Every Mismatch

Payments that don't match cleanly go to the reasoning agent. It categorizes each exception — short pay, deduction, wrong entity, compliance hold, FX variance, duplicate — and recommends an action: post, write off, hold, escalate, or request remittance. It shows its full reasoning for every decision.

MismatchReasoningAgent
5

Generate ERP Posting Entries

The final agent converts all matched and resolved payments into GL journal entries formatted for SAP, Oracle, or NetSuite. Each entry includes the debit/credit accounts, cost center, posting date, and a human-readable narrative. Flagged items are separated into a work queue for human review.

CashPostingAgent
The Pipeline
5 Specialized AI Agents
Each agent does one job well and hands structured output to the next. No monolithic prompt — each specialist runs on the model best suited to its task.
🏦
Agent 1
Bank Statement Intelligence
gpt-4o-mini

Parses raw bank statement JSON, normalizes truncated payer names, extracts remittance references from wire memos, and flags anomalies before any matching begins.

  • Normalized transaction list with resolved payer names
  • Flags: NSF, duplicate, post-dated, stale check, FX, SWIFT truncation
  • Extracted invoice references from payment memos
📒
Agent 2
Open AR Ledger
gpt-4o-mini

Reads the AR ledger and enriches every open invoice with aging analysis, collection risk scores, dispute status, credit hold flags, and factoring arrangements.

  • Invoices enriched with aging buckets (current, 30, 60, 90+ days)
  • Risk flags: disputed, in-litigation, credit hold, OFAC screening
  • Factoring and intercompany relationship map
⚖️
Agent 3
Reconciliation Engine
gpt-4o + Code Interpreter

The core matching engine. Uses fuzzy name matching and reference detection to link payments to invoices, then invokes Code Interpreter to run Python arithmetic for exact verification.

  • Matched, partial-match, and unmatched transaction classifications
  • Python-verified arithmetic for multi-invoice splits and FX amounts
  • Confidence scores and match reasoning for every transaction
🧠
Agent 4
Mismatch Reasoning
gpt-4o

Takes every exception from the reconciliation engine and reasons through it. Categorizes the mismatch type, identifies root cause, and recommends the correct action with full explainability.

  • Exception category: deduction, dispute, short pay, FX variance, wrong entity
  • Recommended action: post, write-off, hold, escalate, request remittance
  • Plain-English reasoning narrative for each decision
✅
Agent 5
Cash Posting
gpt-4o

Converts all reconciled payments into ERP-ready GL journal entries. Separates clean postings from work queue items requiring human review. Produces a complete audit trail with debit/credit accounts, cost centers, and posting narratives.

  • GL journal entries: debit AR control, credit cash, with cost center and posting date
  • Work queue: flagged items needing human review with recommended action pre-filled
  • Summary report: total posted, total held, total exceptions, DSO impact estimate
Edge Cases
Exception Categories Handled
The pipeline handles every exception type that AR teams encounter in real enterprise environments — the same scenarios covered across 10 sample datasets.

💱 FX & Multi-Currency

  • EUR, GBP, CAD payments against USD invoices
  • FX rate verification via Code Interpreter
  • Rounding variance write-off rules

🏢 Entity & Identity

  • SWIFT name truncation (35-char limit)
  • DBA and trade name mismatches
  • Post-acquisition name changes
  • Parent paying for subsidiary

💰 Amount Discrepancies

  • Early-pay discounts (authorized & unauthorized)
  • Freight and damage deductions
  • Short pays and overpayments
  • Multi-invoice consolidated payments

🔗 Third-Party Payments

  • Factoring company remittances
  • Intercompany netting arrangements
  • Cross-border entity structures
  • Third-party payment processors

⏰ Timing Issues

  • Post-dated checks
  • Stale checks (>180 days)
  • NSF (non-sufficient funds) returns
  • Prepayments against future invoices

🔴 Compliance & Legal

  • OFAC sanctions screening triggers
  • Disputed invoices in litigation
  • Wrong legal entity posting risk
  • Payment against bankruptcy-stayed invoices

Technology
How It's Built
Built on Azure AI Foundry with a FastAPI backend and Next.js frontend. All agent outputs stream to the browser in real-time via Server-Sent Events.
🧠
Azure AI Foundry
Assistants API with Code Interpreter for the Reconciliation agent. GPT-4o and GPT-4o-mini for reasoning and parsing agents.
⚡
FastAPI + SSE
Python backend with async streaming via Server-Sent Events. Keepalive pump prevents Railway proxy timeouts during long agent runs.
🌐
Next.js + Vercel
React frontend with real-time SSE rendering. Each agent streams tokens as they arrive — no waiting for the full pipeline to finish.
