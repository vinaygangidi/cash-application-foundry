Section 01
Problem Statement
Accounts Receivable (AR) cash application is one of the most manual, error-prone, and costly processes in enterprise finance. Every day, companies receive hundreds of bank deposits that must be manually matched to open invoices, a process that hasn't fundamentally changed in 30 years.
🔴 The Pain, What AR Teams Face Daily
  • Analysts spend 2–4 hours per batch manually matching bank deposits to invoices in ERP systems
  • Bank payer names are truncated, abbreviated, or changed due to SWIFT 35-char limits, DBA names, and post-acquisition renames
  • Customers pay wrong amounts, early-pay discounts taken late, freight deductions, short pays, damage claims
  • Payments arrive from wrong entities, parent companies paying for subsidiaries, factoring agents, intercompany transfers
  • OFAC/sanctions screening must be done manually, a missed flag is a compliance incident
  • Disputed invoices and stale checks get accidentally applied, creating downstream ERP errors
  • Exception workqueues sit for 3–5 days without SLA assignment, bloating DSO metrics
🟡 The Business Impact
  • Each additional day of cash posting delay = 1 DSO day. For a $500M revenue company, 1 DSO day = ~$1.4M tied up in working capital
  • 3–8% manual error rate on exception transactions creates downstream ERP reconciliation debt
  • AR analyst time spent on cash application averages 40% of total AR headcount cost
  • Missed compliance holds (OFAC, legal) create regulatory and reputational risk
  • No systematic exception categorization means root cause analysis is impossible, the same errors recur every period
  • Inability to auto-post exceptions blocks month-end close acceleration

Section 03
Multi-Model Strategy: Right Model for Each Task
The defining architectural decision: we don't use one model for everything. We route each agent to the Azure OpenAI model whose capabilities match the cognitive complexity of the task.
Agent Model Why This Model Cognitive Task Cost Profile
🏦 BankStatementIntelligenceAgent GPT-4o-mini Structured text extraction, no deep reasoning needed. Mini model is 60% cheaper and 3x faster. Parse payer names, detect anomaly flags, output structured JSON Low
📒 ARLedgerAgent GPT-4o-mini Building a lookup index from structured JSON, deterministic, no nuance required. Build customer index, alias registry, legacy invoice map Low
⚖️ ReconciliationAgent GPT-4o Needs CodeInterpreterTool for exact arithmetic. Complex 8-tier matching logic with pre-checks. Run Python for multi-invoice sums, FX math, discount calc Medium
🧠 MismatchReasoningAgent o3-mini The hardest task: classify 24 exceptions into 7 categories, assess risk tier, reason about contract terms, business rules, and deduction legitimacy. This is where a reasoning model is worth the cost. Deep business judgment: "Is this a legitimate freight deduction or unauthorized short pay?" High (justified)
✅ CashPostingAgent GPT-4o Complex output generation: 12 posting rules, GL account assignments, SLA-based workqueue routing across 4 teams. Generate ERP-ready GL entries and tiered workqueue items Medium
💡 Why This Matters for Production
Running all 5 agents on GPT-4o would cost ~4x more per batch and be slower. Our routing saves ~60% on the two extraction agents (1 & 2), which run on every batch. The o3-mini spend on MismatchReasoningAgent is targeted, it only runs on the 20–30% of transactions that have exceptions, and the reasoning quality directly prevents costly ERP errors. This is production-grade cost optimization, not a prototype shortcut.

GPT-4o

Section 06
How Code Interpreter Works Here vs Using the Foundry UI
Code Interpreter is not just a chat feature. In this pipeline it is the difference between a model that guesses arithmetic and one that actually runs Python and reads the result. This section explains what it does, why it matters for cash application, and how it differs from typing a prompt in the Azure AI Foundry playground.
When you type a prompt in the Foundry UI playground
No Code Interpreter attached
"A customer paid $29,250. They owe invoices for $8,750, $6,500, and $14,000. Does this payment cover all three?"
The model reads the numbers, does the addition inside its own token prediction process, and responds with something like "Yes, 8750 + 6500 + 14000 = 29250, the payment matches."

The problem: The model is not calculating. It is predicting the most likely next token based on training data. Large language models are statistically unreliable on arithmetic, especially when multiple numbers are involved or amounts are close. A $29,400 payment against a $29,250 sum could easily be reported as a match.
Result: The model guesses. In finance, a wrong guess means a misposting. You cannot audit a guess.
With Code Interpreter attached to the Reconciliation Agent
Running in this pipeline
The model does not calculate mentally. Instead, it writes a short Python script and submits it for execution. Azure AI Foundry runs the code in a sandboxed container and returns the printed output. The model reads that output and includes it in its reasoning.
# Model writes this
invoices = [8750, 6500, 14000]
payment = 29250
total = sum(invoices)
print(f"Sum: {total}")
print(f"Match: {total == payment}")

# Azure executes it, returns:
Sum: 29250
Match: True
Result: The model knows. The match decision is based on a Python interpreter output, not token prediction. Fully auditable.

The 5 Calculations Code Interpreter Runs in This Pipeline
Multi-invoice bundle check
invoices = [8750.00, 6500.00, 14000.00]
payment = 29250.00
print(sum(invoices) == payment) # True
Why it matters: a customer bundling 3 invoices in one wire is one of the most common exception types. Without exact verification, the model might accept a $29,400 payment as matching a $29,250 bundle.
Early-pay discount validation
invoice = 30000.00
discount_pct = 2.0
payment = 29400.00
expected = invoice * (1 - discount_pct/100)
print(round(expected, 2) == payment) # True
Why it matters: distinguishes a valid 2% early-pay discount ($600 off $30,000) from an unauthorized short pay. The contractual discount must match exactly.
FX rate verification
eur_amount = 20000.00
fx_rate = 1.125
usd_received = 22500.00
expected_usd = eur_amount * fx_rate
print(abs(expected_usd - usd_received) <= 1.00) # True
Why it matters: a SWIFT payment arrives in EUR. The bank converts it to USD. Code Interpreter verifies the conversion was applied at a reasonable rate, flagging FX_RATE_MISMATCH if the delta exceeds $1.
Stale check age check
from datetime import date
check_date = date(2024, 10, 15)
statement_date = date(2025, 5, 20)
age = (statement_date - check_date).days
print(age > 180) # True - stale check
Why it matters: a check more than 180 days old cannot legally be negotiated at most banks. Depositing it anyway creates a returned item. Code Interpreter catches this before it becomes a problem.
Intercompany netting verification
our_receivable = 150000.00 # what they owe us
our_payable = 120000.00 # what we owe them
payment_received = 30000.00
expected_net = our_receivable - our_payable
print(expected_net == payment_received) # True - valid net settlement
Why it matters: intercompany netting means two related companies settle their mutual AR and AP in a single net payment. Without verifying the math, the model might accept a wrong net amount or misclassify it as a short pay.
The Core Difference in Plain Terms
Foundry UI playground (no Code Interpreter)
You paste your data, the model reads it and produces an answer. All arithmetic happens inside the transformer's forward pass. It is the same process that predicts the next word in a sentence. It is fast and often correct, but it cannot be verified and it fails on edge cases with large or similar-looking numbers. Good for exploration. Not acceptable for posting GL entries.
This pipeline with Code Interpreter on ReconciliationAgent
The model decides what to calculate and writes the code, but a Python interpreter does the actual math. The model never computes a sum, it only reads a printed result. This is a clean separation between reasoning (what to check) and computation (the actual check). Every match decision has a verifiable code execution behind it, not a token prediction.

CREATE TABLE runs (

);

Section 10
Security Architecture
Financial data and compliance decisions require defense-in-depth. Every layer from user login to LLM prompt to database is secured using Azure-native controls.
🔑 Identity & Access, Microsoft Entra ID
Authentication: MSAL + Entra ID SSO. No username/password stored, delegated to enterprise IdP.

RBAC Roles:
• AR_VIEWER, Read results only
• AR_ANALYST, Approve/reject workqueue + run new batches
• AR_MANAGER, Override AI decisions + export GL entries
• COMPLIANCE_OFFICER, Access OFAC holds + legal escalations
• ADMIN, Manage tenant config, model routing, thresholds

Service Accounts: All Azure services use Managed Identity, no API keys or client secrets stored anywhere.
🔐 Data Protection
At Rest: Azure Storage encryption (AES-256) + PostgreSQL Transparent Data Encryption. Customer-managed keys via Azure Key Vault HSM for regulated industries.

In Transit: TLS 1.3 enforced on all endpoints. Azure Front Door + WAF for the public UI. Private endpoints for all backend Azure services (no public internet exposure).

LLM Data Privacy: Azure OpenAI processes data within your Azure tenant, data does NOT leave your subscription. Model is not retrained on your financial data. All prompts include a "do not store" policy header.

PII Minimization: Payer names are hashed before logging in non-production environments. Customer IDs used instead of company names in agent prompts where possible.
🔴 Compliance Controls
OFAC Screening: Every payer name is screened against OFAC SDN list before any matching. Positive match → COMPLIANCE_HOLD, zero LLM processing, immediate escalation. No false negative tolerance.

Segregation of Duties: The agent that generates a GL posting (CashPostingAgent) cannot also approve it. Human approval required for any transaction >$10,000. CRITICAL items require manager-level role.

SOX Controls: All AI decisions are logged with full evidence chain. No GL posting occurs without a run_id traceable to original source document. Annual model accuracy audit report auto-generated.

Regulatory: GDPR-compliant data retention and deletion. CCPA opt-out flags respected. EU customers: data processed in Azure West Europe region only.
🛡 Network & Infrastructure Security
Network Isolation: All Azure resources deployed in a dedicated VNet. PostgreSQL, Blob Storage, Service Bus, and Key Vault accessed only via Private Endpoints, no public internet exposure.

WAF + DDoS: Azure Front Door with WAF policy (OWASP 3.2) in front of the Next.js frontend. Azure DDoS Protection Standard on VNet.

Secrets: Zero secrets in code or environment variables in production. All credentials stored in Azure Key Vault, accessed via Managed Identity. Secret rotation automated with 90-day policy.

Monitoring: Microsoft Defender for Cloud enabled. Azure Monitor alerts on: failed auth attempts, unusual data volumes, COMPLIANCE_HOLD spike, agent failure rate >5%.
