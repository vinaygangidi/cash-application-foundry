Section 01
Problem Statement
Accounts Receivable (AR) cash application is one of the most manual, error-prone, and costly processes in enterprise finance. Every day, companies receive hundreds of bank deposits that must be manually matched to open invoices — a process that hasn't fundamentally changed in 30 years.
🔴 The Pain — What AR Teams Face Daily
  • Analysts spend 2–4 hours per batch manually matching bank deposits to invoices in ERP systems
  • Bank payer names are truncated, abbreviated, or changed due to SWIFT 35-char limits, DBA names, and post-acquisition renames
  • Customers pay wrong amounts — early-pay discounts taken late, freight deductions, short pays, damage claims
  • Payments arrive from wrong entities — parent companies paying for subsidiaries, factoring agents, intercompany transfers
  • OFAC/sanctions screening must be done manually — a missed flag is a compliance incident
  • Disputed invoices and stale checks get accidentally applied, creating downstream ERP errors
  • Exception workqueues sit for 3–5 days without SLA assignment, bloating DSO metrics
🟡 The Business Impact
  • Each additional day of cash posting delay = 1 DSO day. For a $500M revenue company, 1 DSO day = ~$1.4M tied up in working capital
  • 3–8% manual error rate on exception transactions creates downstream ERP reconciliation debt
  • AR analyst time spent on cash application averages 40% of total AR headcount cost
  • Missed compliance holds (OFAC, legal) create regulatory and reputational risk
  • No systematic exception categorization means root cause analysis is impossible — the same errors recur every period
  • Inability to auto-post exceptions blocks month-end close acceleration

Section 03
Multi-Model Strategy — Right Model for Each Task
The defining architectural decision: we don't use one model for everything. We route each agent to the Azure OpenAI model whose capabilities match the cognitive complexity of the task.
Agent Model Why This Model Cognitive Task Cost Profile
🏦 BankStatementIntelligenceAgent GPT-4o-mini Structured text extraction — no deep reasoning needed. Mini model is 60% cheaper and 3x faster. Parse payer names, detect anomaly flags, output structured JSON Low
📒 ARLedgerAgent GPT-4o-mini Building a lookup index from structured JSON — deterministic, no nuance required. Build customer index, alias registry, legacy invoice map Low
⚖️ ReconciliationAgent GPT-4o Needs CodeInterpreterTool for exact arithmetic. Complex 8-tier matching logic with pre-checks. Run Python for multi-invoice sums, FX math, discount calc Medium
🧠 MismatchReasoningAgent o3-mini The hardest task: classify 24 exceptions into 7 categories, assess risk tier, reason about contract terms, business rules, and deduction legitimacy. This is where a reasoning model is worth the cost. Deep business judgment: "Is this a legitimate freight deduction or unauthorized short pay?" High (justified)
✅ CashPostingAgent GPT-4o Complex output generation: 12 posting rules, GL account assignments, SLA-based workqueue routing across 4 teams. Generate ERP-ready GL entries and tiered workqueue items Medium
💡 Why This Matters for Production
Running all 5 agents on GPT-4o would cost ~4x more per batch and be slower. Our routing saves ~60% on the two extraction agents (1 & 2), which run on every batch. The o3-mini spend on MismatchReasoningAgent is targeted — it only runs on the 20–30% of transactions that have exceptions, and the reasoning quality directly prevents costly ERP errors. This is production-grade cost optimization, not a prototype shortcut.

Section 05
Agent-by-Agent Design — Inputs, Outputs & Why Separate
🏦
Agent 1 — BankStatementIntelligenceAgent
GPT-4o-mini
INPUT
Raw bank statement JSON — 35 transactions with payer_raw, remittance_text, amount, payment_type, currency, check_date
OUTPUT
Normalized payment records with 13 anomaly flags, payer alias lookup signals, SWIFT truncation flags, check stale/postdate flags
WHY SEPARATE?
Isolates all noisy text normalization before any invoice matching begins. Mini model handles this cheaply — no business logic reasoning needed here.
📒
Agent 2 — ARLedgerAgent
GPT-4o-mini
INPUT
Open AR JSON — 38 invoices, credit memos, payer_alias_registry (6 aliases), parent_child_hierarchy, intercompany_netting, compliance_config
OUTPUT
customer_index keyed by normalized name, invoice lookup map, legacy_invoice_map, do_not_auto_apply list, compliance_flags dict
WHY SEPARATE?
Builds all lookup structures once so ReconciliationAgent runs pure matching. Separation means AR structure changes don't touch matching logic.
⚖️
Agent 3 — ReconciliationAgent
GPT-4o + CodeInterpreterTool
INPUT
Normalized payments (Agent 1) + AR customer index (Agent 2) from shared thread. Runs 8 pre-checks before matching.
OUTPUT
35 match records — match_tier (1-8), confidence_pct, matched_invoices, delta, pre_check_triggered, exception flag
WHY GPT-4o + CODE?
Writes real Python to verify: sum([8750+6500+14000]) == 29250? 0.98 × 30000 == 29400? FX: 20000 EUR × 1.125 == 22500 USD? Exact arithmetic, no hallucination.
🧠
Agent 4 — MismatchReasoningAgent
o3-mini (Reasoning Model)
INPUT
24 exception transactions with full match context from Agent 3. Exception flag = true, pre_check_triggered, delta, match_tier.
OUTPUT
Per exception: exception_type, exception_category_group (7 categories), reasoning text, risk_tier (CRITICAL/HIGH/MEDIUM/LOW), suggested_gl_code, escalation_contact, sla_hours
WHY o3-mini?
This requires genuine business judgment: contract terms, deduction legitimacy, compliance risk, customer relationship context. o3-mini's chain-of-thought reasoning dramatically outperforms standard models here.
✅
Agent 5 — CashPostingAgent
GPT-4o
INPUT
Full thread: all agent outputs. 35 reconciliation matches + 24 exception analyses with risk tiers and GL recommendations.
OUTPUT
27 GL posting instructions (debit/credit/account), 19 workqueue items (CRITICAL→LOW, team-routed), cash_application_summary with held/posted/deduction breakdowns
WHY SEPARATE?
Separates "think" (Agents 1-4) from "act" (Agent 5). All ERP action generation is isolated — easy to swap ERP targets (SAP vs Oracle vs NetSuite) without touching reasoning logic.
